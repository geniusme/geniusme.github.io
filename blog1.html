Looping Calendaring with PHP

In one of our more recent web development projects, we were creating a training calendaring application with the following criterion:

Training programs may run any given length (i.e. ten days), however if a trainer scheduled a training program from july 1st to july 28th, the program would need to 'loop', so that within the context of the 28 day long period, you would have 2 and 4/5 iterations of a specific training program length. A program can be infinitely long or small, and run for any days specified.

I started by breaking down the resources i would need and grabbing them from my database.


using two simple contstants i can perform the operation necessary to get my desired results:

$days_in_between
//is number of days between booking start date and the date passed from the calendar on the date clicked on the calendar
$days_in_tp_original
//is the number of days that the program was meant to run.

first i need to see if the length of the program on the day clicked, is over the 'loop' limit

if($days_in_between <=$days_in_tp_original){
$query_this_day = $days_in_between;
}

if it is then we're done! but what if it isn't? we need to create a constant that is a base number (for how many times the value between the start date of the program and the current day shown goes into the amount of days in the original program).

else{
$result_divide = floor($days_in_between / $days_in_tp_original);

fantastic! now all we have to do is multiply the number of whole times 'days in between' goes into 'original days', by the original day in the program. This gives us a number to subtract from our initial number of days between when the program starts, and the day on the calendar that was clicked.

$query_this_day = $days_in_between-($result_divide*$days_in_tp_original);

Seems easy enough, but oh snap! there's a problem! Each day in a program is numbered 1-whatever. Using the above math, if we enter into the state of being in the last day of the program (i.e. still a valid number), the result will be 0. This will cause a query error in the database! to deal with this we create the following exception, which speaks for itself.

if($query_this_day==0){$query_this_day=$days_in_tp_original; }}

hopefully this code can help you in your web development pursuits if you are dealing with a similar issue. The whole code snippet of this logic is presented below.

Parsing Dates with Javascript

We currently have a client who has asked us to create an online booking system, so in an attempt to pass most of the parsing client side, i decided to use javascript rather than php to handle looping through our month view for the calendar in question. An interesting bug/twist i came across was how the javascript parse function (i.e. turning a timestamp from 2012/05/22/ into 2398723958723) totally drops the ball on handling the 31st of the month. This is to say that 2012/05/31 returns the exact same integer as 2012/06/01. I circumnavigated this nasty issue by using ajax to only bring up one month view at a time. But lets say that's not an option? what do you do? well thanks to Nicolas Cadou's blog post you can take control away from the javascript parse function and perform the following routine to sanitize this unfortunate javascript short fall.

function makeDateFromIsoString(isoString) {
if (isoString == null)
return null;
var date = new Date();
date.setFullYear(isoString.substr(0, 4));
date.setDate(isoString.substr(8, 2));
date.setMonth(isoString.substr(5, 2) - 1);
date.setHours(0);
date.setMinutes(0);
date.setSeconds(0);
return date;
}

or if you want an even cleaner solution, simply replace your hashes with slashes.
Date(isoString.replace('-', '/', 'g'))
making this simple replacement will prevent javascript from messing up when you use parse. Hope this helps someone as it helped me.
